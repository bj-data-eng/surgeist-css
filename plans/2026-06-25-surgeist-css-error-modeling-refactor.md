# Surgeist CSS Error Modeling Refactor Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace prose-only CSS parse failures with typed error kinds, preserve declaration-level locations for style-boundary validation failures, and tighten tests that prove symbolic calc values survive CSS consumer paths.

**Architecture:** Keep `surgeist-css` as a strict parser and lowering crate whose front door remains `parse_sheet(input: &str) -> Result<surgeist_style::Sheet>`. Add a public `ErrorKind` contract to classify parser, unsupported CSS, and style-validation failures without forcing callers to parse message strings. Preserve successful lowering directly into `surgeist-style`; do not introduce a CSS-authored sheet model in this pass.

**Tech Stack:** Rust 2024, `cssparser = 0.37.0`, `surgeist-style`, crate-local unit tests in `src/lib.rs`, API artifact generated by `api/generator`.

---

## Review Context

This plan responds to a modeling review against `guidance/surgeist-rust-modeling-guide.md`.

Confirmed problems:

- `Error` is public but only exposes `message`, `line`, and `column`, so callers distinguish unsupported property, invalid selector, unsupported value, syntax, and downstream style validation by parsing prose.
- `StrictRuleParser::parse_block` captures one block-level location and reuses it for every `Declarations::try_insert` validation failure.
- Calc shorthand and gap tests only assert `Length::Calc(_)`, which can miss symbolic term loss in consumer paths.

Non-goals:

- Do not create a public `CssSheet`, public `CssDeclaration`, or authored CSS intermediate model.
- Do not move style-owned validation into `surgeist-css`.
- Do not change `surgeist-style` APIs from this crate.
- Do not add dependencies.

- [ ] **Step 5: Update `from_parse_error` and `basic_error_message` to build typed syntax errors**

Change `from_parse_error` and replace `basic_error_message` with `basic_error`:

```rust
fn from_parse_error(error: ParseError<'_, Error>) -> Error {
    match error.kind {
        ParseErrorKind::Custom(error) => error,
        ParseErrorKind::Basic(kind) => basic_error(error.location, kind),

}

fn basic_error(location: cssparser::SourceLocation, kind: BasicParseErrorKind<'_>) -> Error {
    match kind {
        BasicParseErrorKind::EndOfInput => Error::at(
            location,
            ErrorKind::InvalidSyntax {
                reason: "unexpected end of CSS input".to_owned(),
            },
            "unexpected end of CSS input",
        ),
        BasicParseErrorKind::AtRuleInvalid(name) => {
            let name = name.to_string();
            Error::at(
                location,
                ErrorKind::UnsupportedAtRule { name: name.clone() },
                format!("unsupported CSS at-rule `@{name}`"),
            )
        }
        BasicParseErrorKind::QualifiedRuleInvalid => Error::at(
            location,
            ErrorKind::InvalidSyntax {
                reason: "invalid CSS rule".to_owned(),
            },
            "invalid CSS rule",
        ),
        BasicParseErrorKind::AtRuleBodyInvalid => Error::at(
            location,
            ErrorKind::InvalidSyntax {
                reason: "invalid CSS at-rule body".to_owned(),
            },
            "invalid CSS at-rule body",
        ),
        BasicParseErrorKind::UnexpectedToken(token) => {
            let reason = format!("unexpected CSS token `{}`", token.to_css_string());
            Error::at(
                location,
                ErrorKind::InvalidSyntax {
                    reason: reason.clone(),
                },
                reason,
            )
        }
    }
}
```

```rust
fn selector_basic<'i>(error: BasicParseError<'i>) -> ParseError<'i, Error> {
    let location = error.location;
```

- [ ] **Step 8: Add property context for unsupported value errors at the declaration boundary**

In `parse_value`, add a property-name wrapper so nested parser errors with `ErrorKind::UnsupportedValue { property: None, .. }` become property-specific.

Add this helper:

```rust
fn with_property_context<'i>(
    property_name: &str,
    result: std::result::Result<(Property, Value), ParseError<'i, Error>>,
) -> std::result::Result<(Property, Value), ParseError<'i, Error>> {
    result.map_err(|mut error| {
        if let ParseErrorKind::Custom(css_error) = &mut error.kind {
            if let ErrorKind::UnsupportedValue { property, .. } = &mut css_error.kind {
                if property.is_none() {
                    *property = Some(property_name.to_owned());
                }
            }
        }
        error
    })
}
```
