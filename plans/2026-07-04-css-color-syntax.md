# CSS Color Syntax Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Expand strict authored CSS color parsing using pinned `cssparser-color`, while keeping all public color syntax models owned by `surgeist-css`.

**Architecture:** `cssparser-color` is used as the parsing engine for absolute CSS Color 3/4 forms that it already supports: named colors, `transparent`, `currentcolor`, hex with alpha, `rgb()`/`rgba()`, `hsl()`/`hsla()`, `hwb()`, `lab()`, `lch()`, `oklab()`, `oklch()`, and `color(...)`. `surgeist-css` maps those results into crate-owned syntax types. System colors, `color-mix()`, and relative color forms are parsed as symbolic authored color syntax with typed envelopes and no evaluation.

**Tech Stack:** Rust 2024, `cssparser = 0.37.0`, `cssparser-color = 0.5.0`, crate-local parser modules and tests.

## Current State

- `CssColor` is a small RGBA struct with `BLACK`, `WHITE`, and `TRANSPARENT`.
- `parse_color` accepts `#rgb`, `#rrggbb`, `transparent`, `black`, and `white`.
- Color-bearing properties use `CssValue::Color(CssColor)`, `CssValue::TextDecorationColor(CssColor)`, `CssValue::OutlineColor(CssColor)`, and color fields inside border/outline/shadow/text-decoration models.
- Supported declarations containing `var(...)` currently become `CssValue::VariableDependent(...)` before property-specific parsing. Preserve that behavior in this pass for plain `var(...)` color values.

## Target Behavior

Accept as structured `CssColor` syntax:

```css
color: red;
color: rebeccapurple;
color: transparent;
color: currentcolor;
color: #abcd;
color: #11223344;
color: rgb(255 0 0 / 50%);
color: rgba(255, 0, 0, 0.5);
color: hsl(120deg 100% 25% / 0.75);
color: hwb(90 10% 20% / 0.8);
color: lab(50% 20 -30 / 0.9);
color: lch(60% 40 120deg);
color: oklab(0.6 0.1 -0.1);
color: oklch(0.7 0.2 240deg / 80%);
color: color(display-p3 0.8 0.2 0.1 / 0.9);
color: color(display-p3-linear 0.8 0.2 0.1 / 0.9);
color: CanvasText;
color: color-mix(in oklch, red 40%, blue);
color: rgb(from red r g b / alpha);
```

Reject strictly:

```css
color: rgb(1, 2 3);
color: color(unknown-space 1 0 0);
color: color-mix();
color: color-mix(in oklch, red);
color: color-mix(in unknown, red, blue);
color: rgb(from red r g);
color: rgb(from red r g b /);
color: MadeUpSystemColor;
```

## Modeling Rules

- Do not expose `cssparser_color::Color` in the public API.
- Keep `CssColor` crate-owned and inspectable.
- Keep absolute and symbolic color phases explicit:
  - absolute parsed color notation is `CssColor::Rgba`, `CssColor::Hsl`, `CssColor::Hwb`, `CssColor::Lab`, `CssColor::Lch`, `CssColor::Oklab`, `CssColor::Oklch`, or `CssColor::ColorFunction`
  - `currentcolor` is `CssColor::CurrentColor`
  - system colors are `CssColor::System(CssSystemColor)`
  - `color-mix()` is `CssColor::ColorMix(CssColorMix)`
  - relative colors are `CssColor::Relative(CssRelativeColor)`
- Do not evaluate, normalize to renderer color, resolve `currentcolor`, resolve system colors, run color interpolation, substitute variables, or convert color spaces in this crate.
- Preserve existing `CssColor::BLACK`, `CssColor::WHITE`, and `CssColor::TRANSPARENT` associated constants for compatibility where practical.
- Existing `r()`, `g()`, `b()`, and `a()` accessors should be removed or replaced by explicit `as_rgba()`/`CssRgbaColor` accessors. Do not pretend every color is RGBA.
- Existing declarations containing `var(...)` continue to parse as `CssValue::VariableDependent` before property-specific color parsing in this pass. Component-level variable references inside `CssColorComponentExpression` therefore remain empty until a future explicit variable-aware color parsing pass.

fn parse_color_inner<'i, 't>(
    input: &mut Parser<'i, 't>,
) -> std::result::Result<CssColor, ParseError<'i, Error>> {
    if let Ok(color) = input.try_parse(parse_relative_color) {

        return Ok(color);
    }
    Err(unsupported_value(
        input,
        None,

    ))
}
